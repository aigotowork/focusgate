# FocusGate 多语言方案草案

## 目标

这份文档用于描述 FocusGate / 守界 后续的全站多语言体系。当前只做方案设计，不改运行时代码。

目标行为：

- 用户可以在设置页手动选择应用语言。
- 用户未选择语言时，默认跟随浏览器语言偏好，例如 `navigator.languages`。
- Chrome/Edge 扩展自己的名称、描述、按钮标题等元数据使用浏览器官方 `_locales` 机制。
- 应用内部页面使用一套类型安全的运行时语言包，覆盖 React 页面、MV3 service worker、content reminder 和默认规则组模板。

相关源头：

- 产品与命名边界：`prd.md`
- 视觉和文案语气：`DESIGN.md`
- 运行时架构：`project-docs/architecture.md`
- 共享模型源头：`src/shared/types.ts`
- Chrome 扩展 i18n API：https://developer.chrome.com/docs/extensions/reference/api/i18n
- Chrome 扩展本地化指南：https://developer.chrome.com/docs/extensions/how-to/ui/localization
- 浏览器语言偏好：https://developer.mozilla.org/docs/Web/API/Navigator/languages

## 总体建议

建议做“两层 i18n”。

第一层是 Chrome 扩展元数据层：

- 使用 `public/_locales/<locale>/messages.json`。
- 在 `public/manifest.json` 里使用 `__MSG_<key>__` 占位符。
- 只负责扩展名称、短名称、描述、action title 等浏览器管理的文案。

第二层是应用运行时层：

- 在 `src/shared/i18n/` 下维护类型安全的 TypeScript 语言包。
- 所有运行时界面使用同一个语言解析器。
- 用户的手动语言选择存入现有 `AppSettings`。
- 在没有手动选择时，DOM 页面优先读取 `navigator.languages`，扩展上下文可以降级使用 `chrome.i18n.getAcceptLanguages()` 或 `chrome.i18n.getUILanguage()`。

不建议只用 `chrome.i18n.getMessage()` 管整个应用 UI。它适合扩展元数据和简单字符串，但对 React 组合、类型约束、默认模板对象、格式化函数和单元测试都不够顺手。

## 支持语言模型

先保持一个小而明确的支持矩阵：

```ts
export const SUPPORTED_LOCALES = ["zh-CN", "en"] as const;
export type SupportedLocale = (typeof SUPPORTED_LOCALES)[number];

export type LanguagePreference = "auto" | SupportedLocale;
export const DEFAULT_LOCALE: SupportedLocale = "zh-CN";
```

建议第一批语言：

| Locale | 角色 | 说明 |
|---|---|---|
| `zh-CN` | 默认语言，也是当前源文案语言 | 保留现在温和、低刺激的中文文案。 |
| `en` | 第一门国际语言 | 英文界面建议以 `FocusGate` 为主品牌；中文里的意象和口吻需要重写，不要逐字翻译。 |

`zh-TW`、`ja` 等语言可以后续再加。新增语言前最好先确认产品文案、商店资料和测试都能长期维护。

## 语言解析规则

推荐解析顺序：

1. 如果 `settings.language.preference !== "auto"`，使用用户手动选择的语言。
2. 在 DOM 上下文中读取 `navigator.languages`。
3. 在扩展上下文中读取 `chrome.i18n.getAcceptLanguages()`。
4. 再降级读取 `chrome.i18n.getUILanguage()`。
5. 最后使用 `DEFAULT_LOCALE`。

语言匹配需要做大小写归一，并从精确匹配降级到语言大类：

| 浏览器语言 | 解析结果 |
|---|---|
| `zh-CN` | `zh-CN` |
| `zh-Hans-CN` | `zh-CN` |
| `zh` | `zh-CN` |
| `en-US` | `en` |
| `en-GB` | `en` |
| 不支持的语言 | `zh-CN` |

核心解析函数应该保持纯函数，方便测试：

```ts
export function resolveLocale(input: {
  preference: LanguagePreference;
  browserLanguages: readonly string[];
  supportedLocales?: readonly SupportedLocale[];
  fallbackLocale?: SupportedLocale;
}): SupportedLocale;
```

不同运行环境只负责把语言来源传进去：

- React 页面：`navigator.languages`
- Content script：`navigator.languages`
- Background service worker：`chrome.i18n.getAcceptLanguages()` 或 `chrome.i18n.getUILanguage()`
- 单元测试：显式传入语言数组

## 设置数据结构

在 `AppSettings` 里增加语言设置：

```ts
export interface AppLanguageSettings {
  preference: LanguagePreference;
}

export interface AppSettings {
  ruleGroups: RuleGroup[];
  unlocks: UnlockSession[];
  onboardingCompleted: boolean;
  remindedSessionIds: string[];
  pauseUntil?: string;
  language: AppLanguageSettings;
}
```

默认值：

```ts
language: {
  preference: "auto"
}
```

存储迁移只需要给旧数据补上 `language.preference = "auto"`。正常读取设置时不应该顺手改写规则组文案，否则会把用户已经编辑过的内容覆盖掉。

## 运行时语言包形态

应用代码建议用嵌套对象，而不是全局扁平 key。这样更容易按页面和业务域维护。类型定义一份，每个语言包都必须满足同一个接口。

```ts
export interface LocaleCatalog {
  brand: {
    name: string;
    fullName: string;
    slogan: string;
    shortDescription: string;
  };
  common: {
    save: string;
    saved: string;
    cancel: string;
    close: string;
    settings: string;
  };
  options: {
    title: string;
    onboardingTitle: string;
    language: {
      label: string;
      auto: string;
      zhCN: string;
      en: string;
    };
  };
  popup: {
    openSettings: string;
    addToRuleGroup: (groupName: string) => string;
  };
  block: {
    interceptedSite: (site: string) => string;
    unlockReasonLabel: string;
    unlockReasons: Record<UnlockReason, string>;
    temporaryUnlock: (minutes: number, remainingLabel: string) => string;
  };
  reminder: {
    dismiss: string;
    startsSoon: (groupName?: string) => string;
    entersRestrictedTimeAfter: (duration: string) => string;
  };
}
```

中文语言包示意：

```ts
export const zhCN: LocaleCatalog = {
  brand: {
    name: "守界",
    fullName: "FocusGate / 守界",
    slogan: "该专注时，守住你的注意力边界。",
    shortDescription: "用规则组为睡眠、工作和戒断场景建立可执行的网站边界。"
  },
  common: {
    save: "保存设置",
    saved: "已保存",
    cancel: "取消",
    close: "关闭",
    settings: "设置"
  }
};
```

遇到有语序差异的句子，优先使用函数，而不是字符串拼接。例如中文可以说 `加入到 ${groupName}`，英文可能要说 `Add to ${groupName}`。函数能让每种语言自己决定表达顺序。

## React 页面接入方式

React 入口可以加一个很轻的 provider：

```tsx
const I18nContext = createContext<I18nContextValue | undefined>(undefined);

export function I18nProvider(props: { settings: AppSettings; children: ReactNode }) {
  const locale = resolveLocale({
    preference: props.settings.language.preference,
    browserLanguages: navigator.languages
  });
  return (
    <I18nContext.Provider value={{ locale, t: catalogs[locale] }}>
      {props.children}
    </I18nContext.Provider>
  );
}

export function useI18n(): I18nContextValue {
  const value = useContext(I18nContext);
  if (!value) {
    throw new Error("useI18n must be used inside I18nProvider");
  }
  return value;
}
```

当前这些页面本来就会异步读取设置，接入时可以在 settings load 完之后解析语言，再渲染页面。

一旦 i18n 基础落地，新的用户可见文案就不要继续散落在组件局部常量里。可复用文案放进 catalog；默认规则组、默认阻断页这类初始化文案放进 locale-specific defaults。

## 非 React 上下文

### Background service worker

Background 主要需要本地化通知标题和通知正文。

建议封装一个 helper：

```ts
export async function getRuntimeLocale(settings: AppSettings): Promise<SupportedLocale> {
  const browserLanguages =
    typeof chrome !== "undefined" && chrome.i18n?.getAcceptLanguages
      ? await chrome.i18n.getAcceptLanguages()
      : [];

  return resolveLocale({
    preference: settings.language.preference,
    browserLanguages
  });
}
```

使用时：

```ts
const t = catalogs[locale];
await chrome.notifications.create(id, {
  title: t.reminder.startsSoon(decision.ruleGroupName),
  message: t.reminder.notificationMessage(decision.reminderMinutes)
});
```

### Content reminder

Content script 不适合引入 React context。它读取 settings 后，直接使用同一个 resolver 和 catalog：

```ts
const locale = resolveLocale({
  preference: settings.language.preference,
  browserLanguages: navigator.languages
});
const t = catalogs[locale];
```

Shadow DOM 渲染仍然可以保持现在的命令式写法，只需要把所有可见字符串从代码中搬到 catalog。

## 默认模板和用户文案的边界

FocusGate 里有一批可被用户编辑的规则组文案：

- 规则组名称
- 承诺语
- 阻断页标题
- 阻断页描述
- 主按钮文案
- 承接页标题
- 自定义阻断页 HTML
- 承接页 HTML

这些字段一旦创建出来，就应该视为用户内容。全局切换语言时不要静默覆盖它们。

推荐策略：

1. 新安装时，根据当前解析语言创建默认设置。
2. 新建规则组时，根据当前解析语言创建默认模板。
3. 已存在的规则组继续保留存储里的文案，即使用户切换了应用语言。
4. 以后可以在 Options 里加显式操作，例如“把当前规则组恢复为英文默认文案”，但必须由用户主动触发。

为了支持这个边界，后续可以把静态默认常量改造成 builder：

```ts
export function createDefaultRuleGroup(locale: SupportedLocale, now = new Date(0)): RuleGroup;
export function createDefaultBlockPage(kind: "sleep" | "focus", locale: SupportedLocale): BlockPageConfig;
```

现有导出可以保留兼容别名：

```ts
export const DEFAULT_RULE_GROUP = createDefaultRuleGroup(DEFAULT_LOCALE);
export const DEFAULT_SETTINGS = createDefaultSettings(DEFAULT_LOCALE);
```

## Manifest 和商店元数据

扩展元数据使用 Chrome 官方 `_locales` 格式。

`public/manifest.json`：

```json
{
  "default_locale": "zh_CN",
  "name": "__MSG_extensionName__",
  "short_name": "__MSG_extensionShortName__",
  "description": "__MSG_extensionDescription__",
  "action": {
    "default_title": "__MSG_actionTitle__"
  }
}
```

`public/_locales/zh_CN/messages.json`：

```json
{
  "extensionName": {
    "message": "守界 - FocusGate"
  },
  "extensionShortName": {
    "message": "守界"
  },
  "extensionDescription": {
    "message": "用规则组为睡眠、工作和戒断场景建立可执行的网站边界。"
  },
  "actionTitle": {
    "message": "守界"
  }
}
```

注意：Chrome `_locales` 文件夹使用下划线，比如 `zh_CN`；应用运行时建议使用 BCP-47 风格，比如 `zh-CN`。中间用一个很小的 mapper 转换即可。

`project-docs/store/` 里的商店资料后续也应该按语言拆分。商店文案不要直接从运行时语言包派生，因为商店标题、短描述、长描述有不同长度限制和营销表达要求。

## 语言选择 UX

建议在 Options 中增加一个紧凑的“偏好”区域，或者放在现有“隐私与数据”附近：

| Value | 中文界面标签 | 行为 |
|---|---|---|
| `auto` | `跟随浏览器` | 每次读取设置时，根据浏览器语言偏好解析。 |
| `zh-CN` | `简体中文` | 强制使用简体中文运行时界面。 |
| `en` | `English` | 强制使用英文运行时界面。 |

用户切换语言后应该：

- 保存 `settings.language.preference`。
- Options 立即重新渲染。
- Popup、block、handoff、content reminder、background notification 在下一次加载或刷新 settings 后使用新语言。
- 不翻译、不覆盖已有规则组里的用户自定义文案。

## 格式化能力

这些内容应该从硬编码字符串迁移到本地化格式化 helper：

- 时长：`10 分钟`、`1 小时`、`1 小时 30 分钟`、`10 min`、`1 hr`
- 次数：`3 次`、`3 times`
- 星期标签：`一`、`二`、`日`、`Mon`、`Tue`、`Sun`
- `src/shared/sites.ts` 里派生出来的状态标题和说明

共享规则层最好不要直接拼用户可见句子，除非函数显式接收 `locale` 或 catalog。

一个可能的重构方向是把状态判断和状态文案拆开：

```ts
export interface PopupPageContext {
  host: string;
  status: PopupPageStatus;
  matchedRuleGroupId?: string;
  matchedRuleGroupName?: string;
  selectedRuleGroupId?: string;
  canAddToSelectedGroup: boolean;
  activeRuleGroupCount: number;
  upcomingRuleGroupCount: number;
}

export function getPopupStatusCopy(context: PopupPageContext, locale: SupportedLocale): {
  label: string;
  detail: string;
};
```

这样 `src/shared/sites.ts` 仍然主要负责规则判断，UI 文案则集中到 i18n 层。

## 实施阶段

建议分四步做，不要一次性大改所有界面。

### 1. 建立 i18n 基础

- 增加 locale 类型、语言包、resolver、测试和 settings 回填。
- 增加 `public/_locales` 和 manifest 占位符。
- 这一阶段可以先不大规模替换所有 UI 文案。

### 2. 本地化共享默认值和格式化

- 把 `DEFAULT_RULE_GROUP`、默认阻断页、品牌文案、时长/次数标签改造成 locale-aware builder 或 helper。
- 给中文和英文默认模板补测试。

### 3. 逐个迁移运行时界面

- 先迁移 Options，因为语言选择入口在这里。
- 再迁移 Popup，因为它面积小但状态文案集中。
- 然后迁移 Block/Handoff，因为它们依赖默认模板和主操作文案。
- 最后迁移 Content reminder 和 Background notification，因为它们是非 React 写法。

### 4. 更新文档和测试

- 新增 `src/shared/i18n/AGENTS.md`。
- 更新 `project-docs/architecture.md`。
- 更新 Playwright smoke tests，覆盖强制 `zh-CN`、强制 `en` 和 `auto` fallback。

## 验证计划

单元测试：

- `resolveLocale` 覆盖精确匹配、语言大类匹配、脚本别名、不支持语言回退。
- 设置迁移会给旧数据补上 `language.preference = "auto"`。
- 每个语言包都满足同一个 TypeScript 接口。
- 默认规则组只在创建时按语言生成，不在语言切换时改写旧数据。
- 格式化 helper 输出预期的时长、次数和星期标签。

E2E 测试：

- Options 语言选择可以保存并立即重渲染。
- Popup 能读取用户选择的语言。
- Block page 内置 UI 随语言变化，但存储里的自定义规则文案保持不变。
- Content reminder 使用用户选择的语言。
- build 产物包含 `_locales` 文件和 `default_locale`。

手动检查命令：

```bash
npm run typecheck
npm test
npm run test:e2e
npm run build
```

然后把 `dist/` 作为 unpacked extension 加载，检查：

- 扩展名称和描述能按浏览器 UI 语言显示。
- 应用内语言选择能覆盖浏览器语言。
- `跟随浏览器` 能根据浏览器语言偏好解析。

## 待确认问题

- 英文界面是否只使用 `FocusGate`，还是保留 `FocusGate / 守界` 作为全球品牌露出？
- 不支持的语言应该继续回退到 `zh-CN`，还是在国际化上架后改成回退到 `en`？
- 新建规则组应该使用“当前解析语言”，还是只看用户显式选择？当前建议使用当前解析语言，包括 `auto` 解析结果。
- 对从未编辑过的默认规则组，是否需要提供“刷新为当前语言默认文案”的显式按钮？
- 第一版要本地化多少 Chrome Web Store 元数据？

## 给后续 Agent 的说明

真正实现这套方案时，应新增 `src/shared/i18n/AGENTS.md`，明确以下规则：

- 语言包必须类型安全、字段完整。
- 运行时 UI 不要再硬编码用户可见文案。
- 不要自动翻译或覆盖已经存储的用户规则组文案。
- Chrome `_locales` 元数据和应用运行时语言包保持分离。
- 每次新增语言、默认模板或格式化规则，都要同步更新测试。
